# Fully Internal / Private Cloud
**Scope:** Applies to all proprietary and regulated data processed by internally hosted AI systems. Ensures model integrity, confidentiality, and resilience against adversarial threats.
## 1. Model Integrity & Trust Controls
-	**Signed Model Artifacts:** All model binaries, weights, and config files must be cryptographically signed using HSM-backed key management.
-	**SBOM Required:** All dependencies and training components must maintain an up to date Software Bill of Materials.
- **Secure Model Registry:** Versioned, access controlled storage with tamper evident logs.
-	**Environmental Boundaries:** Strict separation of training, staging, and production environments.

**Validation Activities:**
-	Hash verification at load time
-	Immutable logging for artifact access
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 2. Input Security & Adversarial Resistance
-	**Input Validation + Normalization:** Prevent injection payloads, adversarial prompt patterns.
-	**Adversarial Detection Layer:** Detect anomalous vectors or adversarial examples.
-	**Rate and Depth Limiting:** Prevent model extraction attacks.
-	**Prompt Safety Policies** (for generative models): Disallow unsafe context shaping.

**Required Testing:**
-	Evasion testing (FGSM/PGD methods)
-	Prompt injection red team testing
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 3. Data Protection & Privacy Controls
-	**Differential Privacy** (required for regulated training data sets)
-	**Data Minimization:** Only relevant data used in training.
-	**Encrypted Feature Store:** AES 256 or stronger at rest.
- **No External Egress from training cluster.**

**Operational Measures:**
-	Pseudonymization/tokenization by default
-	Lineage tracking for sensitive inputs
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 4. Output Safety Controls
•	**Hallucination Monitoring:** Logging unexpected/confident-but-wrong outputs.
<br />•	**Response Filtering:** Match against safety and compliance detection rules.
<br />•	**Explainability for Validation:** Model card must justify data sources and logic.
<br />**Automated Review Signals:**
<br />•	Confidence thresholds
<br />•	Restricted term detection
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 5. Drift Monitoring & Retraining Protection
•	**Continuous Drift Tracking** (inputs + model performance)
<br />•	**Retraining Authorization Workflow:** CAIO + Security sign off required.
<br />•	**Rollback Strategy:** Previous signed artifacts available for instant restore.
<br />**Alert Thresholds:**
<br />•	Feature distribution changes
<br />•	Security deviation triggers quarantine
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 6. Internal Access Governance
-	**Least Privilege RBAC** using Zero Trust enforcement
-	**Just in Time Access** Only
-	**Segregation of Duties:** No single admin can push to production and approve artifacts.

**Monitoring:**
-	Privileged access recorded and reviewed weekly
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 7. Compliance Mapping
-	**NIST AI RMF:** Secure design, measurement, monitoring
-	**ISO/IEC 27001:** Asset mgmt., cryptography, access control
-	**ISO/IEC 42001:** AIMS lifecycle compliance
-	**GDPR:** DP by design, lawful basis for training data
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 8. Mandatory Policy Notice
No proprietary, classified, or regulated data may leave internal AI boundaries without encryption, masking, and pre approved business justification logged in the AI System Registry.

# Hybrid (Internal + 3rd-Party LLMs)
**Scope:** Covers AI systems using both internal models and external cloud-based LLM APIs. Ensures model integrity, data security, and adversarial resilience across both environments.
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 1. Model Integrity & Trust Controls
-	**Signed Internal Model Artifacts:** Internal models must be cryptographically signed; external LLM outputs are monitored for consistency.
-	**Dependency SBOMs:** Track internal and vendor-provided dependencies.
-	**Secure Registry:** Internal models versioned; external outputs logged.
-	**Separation of Environments:** Internal training, inference sandbox, and external API interactions clearly isolated.

**Validation Activities:**
-	Internal model hash checks
-	Logging of external API interactions
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 2. Input Security & Adversarial Resistance
-	**Internal Inputs:** Validate against adversarial patterns.
-	**External Inputs:** Prompt sanitization and tokenization before sending to vendor APIs.
-	**Rate & Depth Limiting:** Prevent model extraction attacks externally.
-	**Safety Policies:** Prohibit sensitive or PII in external API payloads.

**Testing:**
-	Adversarial evasion internal testing
-	External prompt injection tests
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 3. Data Protection & Privacy Controls
-	**Differential Privacy** applied internally where required
-	**Tokenization/Masking** before external LLM queries
-	**Encrypted Feature Store** for internal data
-	**No unencrypted PII leaves internal boundary**

**Operational Measures:**
-	Lineage tracking for internal + external data paths
-	Logging all API calls for auditing
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 4. Output Safety Controls
-	Internal and external model responses filtered for sensitive data
-	Confidence thresholds, anomaly detection, and output consistency checks
-	Monitor external outputs for hallucinations or compliance violations
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 5. Drift Monitoring & Retraining Protection
-	Internal model drift tracking
-	External vendor performance and response consistency monitoring
-	Retraining or model update requires CAIO + DPO approval
-	Rollback plan for internal models; contingency for external LLM fallback
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 6. Internal Access Governance
-	RBAC, least privilege, and JIT access for internal models
-	External API keys rotated regularly
-	Segregation of duties between internal model deployment and external API configuration

**Monitoring:**
-	Weekly audit of privileged access
-	API key usage alerts for anomalies
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 7. Compliance Mapping
-	**NIST AI RMF:** Secure design, monitoring, and risk management
-	**ISO/IEC 27001:** Asset management, cryptography, and access control
-	**ISO/IEC 42001:** AI lifecycle compliance
-	**EU AI Act:** Provider/user obligations for high-risk AI
-	**GDPR:** Data protection measures for external processing
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 8. Policy Language (Modified for Hybrid)
No proprietary, classified, or regulated data may leave internal boundaries to external AI services without encryption, tokenization/masking, and explicit approval recorded in the AI System Registry. Vendor API interactions must be pre-approved, and sensitive data must never be transmitted unencrypted.

# Cloud-Only (SaaS)
Scope: Applies to AI systems that rely entirely on external SaaS AI providers. Ensures model integrity, strict data handling, and adversarial resilience while limiting exposure of proprietary/regulatory data.
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 1. Model Integrity & Trust Controls
-	Internal monitoring for API outputs consistency and anomalies
-	Versioning and logging of all queries and responses
-	Require **vendor-signed data handling and security policy documentation**
-	Vendor SLAs must include encryption standards, retention policies, and breach notification clauses
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 2. Input Security & Adversarial Resistance
-	Mandatory DLP, tokenization, and masking for all data sent externally
-	Strict input validation to prevent adversarial injection or prompt-based data exfiltration
-	Rate and depth limiting to prevent model extraction
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 3. Data Protection & Privacy Controls
-	No unencrypted proprietary or regulated data leaves internal boundary
-	Tokenization or anonymization applied before API requests
-	Logs maintained for all outbound requests with secure retention
-	Continuous verification of vendor compliance with contractually defined data handling policies
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 4. Output Safety Controls
-	Monitor and filter all SaaS AI responses for sensitive data leakage
-	Track hallucinations and non-compliant outputs
-	Implement fallback to offline/internal models for high-risk operations
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 5. Drift Monitoring & Model Update Controls
-	Monitor SaaS API responses for drift and consistency with internal expectations
-	Document and approve any operational changes affecting model behavior
-	Contingency planning for vendor service interruption or compliance failures
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 6. Governance & Access Control
-	RBAC and least-privilege access for managing API keys
-	Frequent rotation and auditing of all SaaS API credentials
-	Segregation of duties between internal users and external interactions
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 7. Compliance Mapping
-	**NIST AI RMF:** Risk management, monitoring, vendor reliance
-	**ISO/IEC 27001:** Asset and access control, vendor management
-	**ISO/IEC 42001:** AI lifecycle governance
-	**EU AI Act & GDPR:** Data transfer, processing, and high-risk AI obligations
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 8. Policy Language (Strict)
**No proprietary, regulated, or sensitive data may be sent to external SaaS AI services without CAIO and DPO approval, documented in the AI System Registry, and only after ensuring vendor-signed data handling policies and security assurances are in place. Unauthorized transmission will be treated as a compliance violation.**
