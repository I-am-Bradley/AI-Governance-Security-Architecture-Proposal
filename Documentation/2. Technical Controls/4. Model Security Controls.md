# Fully Internal / Private Cloud
**Scope:** Applies to all proprietary and regulated data processed by internally hosted AI systems. Ensures model integrity, confidentiality, and resilience against adversarial threats.
## 1. Model Integrity & Trust Controls
•	**Signed Model Artifacts:** All model binaries, weights, and config files must be cryptographically signed using HSM-backed key management.
<br />•	**SBOM Required:** All dependencies and training components must maintain an up to date Software Bill of Materials.
<br />• **Secure Model Registry:** Versioned, access controlled storage with tamper evident logs.
<br />•	**Environmental Boundaries:** Strict separation of training, staging, and production environments.
<br />**Validation Activities:**
<br />•	Hash verification at load time
<br />•	Immutable logging for artifact access
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 2. Input Security & Adversarial Resistance
•	**Input Validation + Normalization:** Prevent injection payloads, adversarial prompt patterns.
<br />•	**Adversarial Detection Layer:** Detect anomalous vectors or adversarial examples.
<br />•	**Rate and Depth Limiting:** Prevent model extraction attacks.
<br />•	**Prompt Safety Policies** (for generative models): Disallow unsafe context shaping.
<br />**Required Testing:**
<br />•	Evasion testing (FGSM/PGD methods)
<br />•	Prompt injection red team testing
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 3. Data Protection & Privacy Controls
•	**Differential Privacy** (required for regulated training data sets)
<br />•	**Data Minimization:** Only relevant data used in training.
<br />•	**Encrypted Feature Store:** AES 256 or stronger at rest.
<br />•	**No External Egress** from training cluster.
<br />**Operational Measures:**
<br />•	Pseudonymization/tokenization by default
<br />•	Lineage tracking for sensitive inputs
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 4. Output Safety Controls
•	**Hallucination Monitoring:** Logging unexpected/confident-but-wrong outputs.
<br />•	**Response Filtering:** Match against safety and compliance detection rules.
<br />•	**Explainability for Validation:** Model card must justify data sources and logic.
<br />**Automated Review Signals:**
<br />•	Confidence thresholds
<br />•	Restricted term detection
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 5. Drift Monitoring & Retraining Protection
•	**Continuous Drift Tracking** (inputs + model performance)
<br />•	**Retraining Authorization Workflow:** CAIO + Security sign off required.
<br />•	**Rollback Strategy:** Previous signed artifacts available for instant restore.
<br />**Alert Thresholds:**
<br />•	Feature distribution changes
<br />•	Security deviation triggers quarantine
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 6. Internal Access Governance
•	**Least Privilege RBAC** using Zero Trust enforcement
<br />•	**Just in Time Access** Only
<br />•	**Segregation of Duties:** No single admin can push to production and approve artifacts.
<br />**Monitoring:**
<br />•	Privileged access recorded and reviewed weekly
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 7. Compliance Mapping
•	**NIST AI RMF:** Secure design, measurement, monitoring
<br />•	**ISO/IEC 27001:** Asset mgmt., cryptography, access control
<br />•	**ISO/IEC 42001:** AIMS lifecycle compliance
<br />•	**GDPR:** DP by design, lawful basis for training data
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 8. Mandatory Policy Notice
No proprietary, classified, or regulated data may leave internal AI boundaries without encryption, masking, and pre approved business justification logged in the AI System Registry.

# Hybrid (Internal + 3rd-Party LLMs)
**Scope:** Covers AI systems using both internal models and external cloud-based LLM APIs. Ensures model integrity, data security, and adversarial resilience across both environments.
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 1. Model Integrity & Trust Controls
•	**Signed Internal Model Artifacts:** Internal models must be cryptographically signed; external LLM outputs are monitored for consistency.
<br />•	**Dependency SBOMs:** Track internal and vendor-provided dependencies.
<br />•	**Secure Registry:** Internal models versioned; external outputs logged.
<br />•	**Separation of Environments:** Internal training, inference sandbox, and external API interactions clearly isolated.
<br />**Validation Activities:**
<br />•	Internal model hash checks
<br />•	Logging of external API interactions
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 2. Input Security & Adversarial Resistance
•	**Internal Inputs:** Validate against adversarial patterns.
<br />•	**External Inputs:** Prompt sanitization and tokenization before sending to vendor APIs.
<br />•	**Rate & Depth Limiting:** Prevent model extraction attacks externally.
<br />•	**Safety Policies:** Prohibit sensitive or PII in external API payloads.
<br />**Testing:**
<br />•	Adversarial evasion internal testing
<br />•	External prompt injection tests
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 3. Data Protection & Privacy Controls
•	**Differential Privacy** applied internally where required
<br />•	**Tokenization/Masking** before external LLM queries
<br />•	**Encrypted Feature Store** for internal data
<br />•	**No unencrypted PII leaves internal boundary**
<br />**Operational Measures:**
<br />•	Lineage tracking for internal + external data paths
<br />•	Logging all API calls for auditing
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 4. Output Safety Controls
•	Internal and external model responses filtered for sensitive data
<br />•	Confidence thresholds, anomaly detection, and output consistency checks
<br />•	Monitor external outputs for hallucinations or compliance violations
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 5. Drift Monitoring & Retraining Protection
•	Internal model drift tracking
<br />•	External vendor performance and response consistency monitoring
<br />•	Retraining or model update requires CAIO + DPO approval
<br />•	Rollback plan for internal models; contingency for external LLM fallback
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 6. Internal Access Governance
•	RBAC, least privilege, and JIT access for internal models
<br />•	External API keys rotated regularly
<br />•	Segregation of duties between internal model deployment and external API configuration
<br />**Monitoring:**
<br />•	Weekly audit of privileged access
<br />•	API key usage alerts for anomalies
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 7. Compliance Mapping
•	**NIST AI RMF:** Secure design, monitoring, and risk management
<br />•	**ISO/IEC 27001:** Asset management, cryptography, and access control
<br />•	**ISO/IEC 42001:** AI lifecycle compliance
<br />•	**EU AI Act:** Provider/user obligations for high-risk AI
<br />•	**GDPR:** Data protection measures for external processing
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 8. Policy Language (Modified for Hybrid)
No proprietary, classified, or regulated data may leave internal boundaries to external AI services without encryption, tokenization/masking, and explicit approval recorded in the AI System Registry. Vendor API interactions must be pre-approved, and sensitive data must never be transmitted unencrypted.

# Cloud-Only (SaaS)
Scope: Applies to AI systems that rely entirely on external SaaS AI providers. Ensures model integrity, strict data handling, and adversarial resilience while limiting exposure of proprietary/regulatory data.
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 1. Model Integrity & Trust Controls
•	Internal monitoring for API outputs consistency and anomalies
<br />•	Versioning and logging of all queries and responses
<br />•	Require **vendor-signed data handling and security policy documentation**
<br />•	Vendor SLAs must include encryption standards, retention policies, and breach notification clauses
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 2. Input Security & Adversarial Resistance
•	Mandatory DLP, tokenization, and masking for all data sent externally
<br />•	Strict input validation to prevent adversarial injection or prompt-based data exfiltration
<br />•	Rate and depth limiting to prevent model extraction
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 3. Data Protection & Privacy Controls
•	No unencrypted proprietary or regulated data leaves internal boundary
<br />•	Tokenization or anonymization applied before API requests
<br />•	Logs maintained for all outbound requests with secure retention
<br />•	Continuous verification of vendor compliance with contractually defined data handling policies
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 4. Output Safety Controls
•	Monitor and filter all SaaS AI responses for sensitive data leakage
<br />•	Track hallucinations and non-compliant outputs
<br />•	Implement fallback to offline/internal models for high-risk operations
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 5. Drift Monitoring & Model Update Controls
•	Monitor SaaS API responses for drift and consistency with internal expectations
<br />•	Document and approve any operational changes affecting model behavior
<br />•	Contingency planning for vendor service interruption or compliance failures
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 6. Governance & Access Control
•	RBAC and least-privilege access for managing API keys
<br />•	Frequent rotation and auditing of all SaaS API credentials
<br />•	Segregation of duties between internal users and external interactions
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 7. Compliance Mapping
•	**NIST AI RMF:** Risk management, monitoring, vendor reliance
<br />•	**ISO/IEC 27001:** Asset and access control, vendor management
<br />•	**ISO/IEC 42001:** AI lifecycle governance
<br />•	**EU AI Act & GDPR:** Data transfer, processing, and high-risk AI obligations
________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________________
## 8. Policy Language (Strict)
**No proprietary, regulated, or sensitive data may be sent to external SaaS AI services without CAIO and DPO approval, documented in the AI System Registry, and only after ensuring vendor-signed data handling policies and security assurances are in place. Unauthorized transmission will be treated as a compliance violation.**
